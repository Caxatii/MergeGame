## Краткая инструкция по использования Domain-Driven-Disign (DDD) в проекте.

Проект делиться на слои **Application**, **Domain**, **Infrastructure**, 
**Presentation** и **ContractInterfaces**. 

### Слои
1. **Domain** - логика доменной модели,
2. **Application** - бизнес-логика,
3. **Infrastructure** - инфраструктура (например, репозитории),
4. **Presentation** - пользовательский интерфейс,
5. **ContractInterfaces** - интерфейсы для взаимодействия между слоями.

### Domain
Наименьшие единицы проекта, описывающие способ взаимодействия с ними.

1. **Entities** - объекты, которые представляют собой состояния и поведение.
2. **Value Objects** - объекты, которые не имеют состояния, а только содержат данные.
3. **Aggregates** - группы связанных объектов, которые представляют собой целые сущности.

### Application
Слой, который реализует взаимодействие с моделями (доменом).

1. Координация бизнес-процессов: управление выполнением use cases
2. Обработка запросов: получение и обработка команд от внешнего слоя
3. Взаимодействие с доменом: организация работы с доменными объектами
4. Управление транзакциями: контроль за целостностью операций

### Infrastructure
Слой, который реазирует взаимодействие данных с внешними сервисами:

1. Хранение данных: реализация механизмов сохранения и извлечения данных
2. Взаимодействие с внешними сервисами: работа с API, очередями сообщений
3. Технические сервисы: логирование, мониторинг, безопасность

### ContractInterfaces
Слой, который является контрактом взаимодействия между слоями приложения. 
Содержит исключительно интерфейсы. Интерфейсы реализуются в других слоях.
